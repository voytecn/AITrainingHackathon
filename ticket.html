Payment OOM causes Stripe charges without local transaction records

P0
Component
payments
Assignee
payments-team
Labels
payments oom data-integrity stripe reconciliation incident

Description

Payment service threw `java.lang.OutOfMemoryError: Java heap space` in `PaymentBatch.aggregate` while handling a charge request (req_id=7f3a9b). The transaction was NOT committed locally, but the user was already charged on Stripe, leaving the system in an inconsistent state requiring manual reconciliation.

This is a data integrity / financial correctness issue: any user hitting this OOM path will be charged externally without a corresponding local record. At least one user (u_4421) is currently affected and flagged for manual reconciliation.

Root cause appears to be unbounded memory growth in `PaymentBatch.aggregate` (PaymentBatch.java:142). Additionally, the charge flow is not atomic with respect to Stripe — the external charge succeeds before the local transaction is committed, so any failure after the Stripe call (OOM, crash, etc.) produces orphaned charges.

Steps to Reproduce

  1. At 2026-05-19T14:32:12.445Z, request req_id=7f3a9b entered PaymentController.charge (PaymentController.java:54).
  2. PaymentService.handleCharge (PaymentService.java:88) invoked PaymentBatch.aggregate (PaymentBatch.java:142).
  3. PaymentBatch.aggregate exhausted the JVM heap and threw java.lang.OutOfMemoryError.
  4. Local transaction commit failed; however the Stripe charge for user u_4421 had already succeeded.
  5. Result: user u_4421 charged on Stripe with no corresponding local transaction record — flagged for manual reconciliation.
