Description

The `search-api` service is experiencing connection pool exhaustion against its database. HikariCP is throwing `PoolInitializationException: Connection timeout after 10000ms` after waiting the full 10s for a free connection.

Pool state at time of incident: `active=20 idle=0 waiting=14` — the pool is fully saturated with 14 additional requests queued. In the last 5 minutes, 234 connection-timeout errors have occurred, and the service is serving stale (6h old) cached fallback results to users.

While the fallback prevents user-facing 5xx errors (requests still return 200), search results are stale and degraded. Root cause is likely one of: (1) a slow/long-running query holding connections, (2) traffic spike exceeding pool capacity, (3) a connection leak in `ProductRepository.findByQuery`, or (4) downstream database performance degradation.

First stack frame: `com.acme.search.repo.ProductRepository.findByQuery(ProductRepository.java:67)` called from `SearchService.search(SearchService.java:42)`.

Steps to Reproduce

  1. User u_8821 issued GET /api/search?q=running+shoes (req_id=a1b2c3) at 2026-05-19T09:14:22Z.
  2. Request acquired a DB connection from the Hikari pool (12/20 in use at that moment).
  3. 10 seconds later, the connection attempt for the query timed out with HikariPool PoolInitializationException.
  4. Pool state immediately after: active=20, idle=0, waiting=14 — fully saturated.
  5. Service fell back to stale cached results (6h old) and returned HTTP 200.
  6. Over the following 5 minutes, 234 similar connection timeouts and fallback responses were observed.
